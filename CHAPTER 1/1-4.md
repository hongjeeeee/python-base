# 1.4 파이썬 프로그래밍 가이드라인
다음은 Tim Peters가 작성한 **"파이썬의 도(The zen of Python)"**이라는 내용이며 "파이썬 다움"이라는 전반적인 개념을 간략하게 소개하고 있다. 
- 코딩의 내용이 가능한 한 정확하게 전달될 수 있어야 한다. 
- 하나의 파이썬 문장은 한 줄에 적는다. 

파이썬에서 `import this`를 실행하면 다음과 같이 "파이썬의 도"라는 글이 출력된다.

```
The Zen of Python, by Tim Peters

Beautiful is better than ugly.
Explicit is better than implicit.
Simple is better than complex.
Complex is better than complicated.
Flat is better than nested.
Sparse is better than dense.
Readability counts.
Special cases aren't special enough to break the rules.
Although practicality beats purity.
Errors should never pass silently.
Unless explicitly silenced.
In the face of ambiguity, refuse the temptation to guess.
There should be one-- and preferably only one --obvious way to do it.
Although that way may not be obvious at first unless you're Dutch.
Now is better than never.
Although never is often better than *right* now.
If the implementation is hard to explain, it's a bad idea.
If the implementation is easy to explain, it may be a good idea.
Namespaces are one honking great idea -- let's do more of those!
```

위의 내용 중에서 몇 가지만 직역해서 적어보면 다음과 같다.
- 지저분하게 하기보다 멋지게 하는 것이 낫다.
- 드러나지 않게 하기보다 드러내는 것이 낫다.
- 복잡하게 하기보다 드러내는 것이 낫다.
- 빽빽하게 적는 그것보다 여유 공간을 주는 것이 낫다.
- 읽기 쉽게 작성하는 것이 옳다.
- 들여쓰기는 다음과 같이 4칸으로 한다

```python
for i in range(5):
  print(i)
```

- 사칙 연산이 이어질 때는 다음과 같이 연산자를 분리한다(good). 두번 째와 같이하면 연산자와 성분이 분리되기 때문에 읽을 때 흐름이 끊어져서 내용을 파악하기에 불편함을 알 수 있다(bad).

```python
# Good
total = name
        + id
        + score
        + age
        + gender

# Bad
total = name +
        id +
        score +
        age +
        gender +
```

- 공백을 주는 방법   
프로그램을 작성할 때 문자이나 표현에서 적절한 공백을 주면 더욱 읽기 쉽게 된다. 그러나 아래의 예와 같이 불필요한 공백은 피하는 것이 좋다. 특별히 `print(x, y)`와 같은 괄호 내에서의 공백은 주지 않는 것이 보기에 좋다.

```python
# Good
x = 3
y = 7
if x > 3:
    print(x, y)

# Bad
x =3
y= 7
if x> 3:
    print( x , y )
```

위에서 언급한 몇 가지 사례들은 파이썬 가이드라인의 일부에 불과하다. 물론 파이썬 프로그래밍을 할 때 가이드라인의 내용을 모두 준수할 수는 없겠으나 가능하면 그대로 따라 주는 것이 보기에도 좋고, 다른 프로그래머들과의 교류에도 도움이 될 수 있다.
